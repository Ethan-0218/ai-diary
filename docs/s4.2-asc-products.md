# S4.2 — ASC 인앱구매 등록 체크리스트 (콘솔 수동 작업)

> **목적**: App Store Connect에 IAP 상품을 만드는 **당신의 콘솔 작업** 가이드. 코드 카탈로그(`packages/shared/src/products.ts`)와 **productId가 1:1로 일치**해야 함.
> **선행**: S4.1 데이터 모델 + 주간 티어 카탈로그 완료(코드). **productId는 ASC에서 생성 후 변경 불가**하니 아래 표 그대로 만들 것.

---

## ⚠️ 0. 상품 타입 = **Consumable**(소비성) — 중요

직관과 달리 **Non-Consumable(비소비성)이 아니라 Consumable**로 만들어야 합니다.
- **이유**: Non-Consumable은 **Apple ID당 평생 1회만** 구매 가능 → "이달의 일기"를 6월에 사면 **7월에 같은 SKU를 다시 못 삼**. 매달 재구매가 핵심인 우리 모델과 충돌.
- **소유(buy-to-own) 감성은 유지됨**: 일기장 소유는 **우리 백엔드**가 책임짐(`Notebook`이 `userId`에 영구 귀속). Apple IAP는 결제 레일일 뿐.
- **"구매 복원"**: StoreKit restore가 아니라 **계정 로그인 = 내 일기장 동기화**(백엔드가 소유 노트북 반환). Apple이 기대하는 "복원" UX는 "계정으로 내 구매 불러오기"로 충족.
- 비유: 종이 일기장을 매달 새로 사는 것 — 각 구매가 별개 객체. 소비성이 맞다.

> ※ 구독(Auto-renewable)은 유저가 1차 제외 결정. Non-renewing subscription도 부적합(시한부 접근 아님). → **Consumable 확정 권장.**

## 1. 등록할 SKU (14개) — productId 그대로

**Bundle ID**: `com.ai-diary.app` / 모든 IAP productId 접두사 `com.ai-diary.notebook.`

### 연대기(월간 기간형) — 라인당 4 주간 티어
가격은 **W4(정가) 기준 비례**(W3≈75% / W2≈50% / W1≈25%). 한 라인의 4 티어는 같은 일기장(그 달 남은 칸)을 발행하고 **가격만 다름**. 백엔드가 구매 시점에 맞는 티어만 띄움.

### 컬렉션(칸형, 단일가) — 기간형보다 비싸게(§4.5-A 유연 프리미엄)
| 상품 | productId | 제안가(₩) |
|---|---|---|
| 30편의 소설 | `com.ai-diary.notebook.novel_30` | 6,600 |
| 30일의 기록 | `com.ai-diary.notebook.plain_30` | 6,600 |

> 제안가는 출발점. 실제는 ASC 가격표(Price Point) 드롭다운에서 가장 가까운 값으로. 라인 내 4티어가 75/50/25%에 정확히 안 떨어지면 가까운 포인트로.

## 2. 각 SKU 작성 항목 (ASC → 기능 → 인앱 구매)
- **유형**: Consumable
- **참조 이름**(내부용): 위 "라인 + 티어"로 (예: `plain month W4`).
- **제품 ID**: 위 표 그대로(오타 주의, 생성 후 변경 불가).
- **가격**: 위 제안가에 가까운 Price Point.
- **현지화(한국어, 1차 언어)**:
  - 표시 이름: 예) "이달의 일기 (한 달)" / "이달의 일기 (남은 3주)" / "이달의 일기 (남은 2주)" / "이달의 일기 (남은 1주)". 칸형: "30편의 소설" / "30일의 기록".
  - 설명: `products.ts`의 description 활용.
- **심사 정보**: 스크린샷(구매 화면) 1장 + 심사 메모(처음 제출 시 필요. S4.3에서 스토어 화면 나오면 캡처).

## 3. 순서·팁
1. 먼저 **W4 4개 + 칸형 2개**(총 6개)만 만들어 S4.3 결제 플로우를 빨리 뚫고, W3/W2/W1(9개)은 그 다음에 채워도 됨(티어 해석은 W4만 있어도 폴백 동작).
2. 로컬 테스트는 ASC 승인 전에도 **Xcode StoreKit Configuration(.storekit)** 으로 가능(S4.3에서 이 productId들로 생성).
3. IAP는 **유료 계약(Paid Apps) Active** 여야 제출 가능 — S1 B5 상태 확인.
4. 첫 제출 시 IAP는 앱 빌드와 **함께 심사**됨(S8). 내부 테스트(S6)+샌드박스(S7)는 승인 전에도 가능.
