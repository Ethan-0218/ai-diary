# AI 일기 Agent v2 — 설계 문서 (living doc)

> PoC에서 배운 것을 바탕으로 일기 agent를 **새로 설계**한다. 이 문서는 같이 채워나가는 살아있는 문서다.
> 상태: **v0.1 초안** — `❓`는 함께 정해야 할 결정, `🔧 PoC 완료`는 PoC에서 이미 검증/구현된 것.

---

## 1. 비전 / 북극성 ✅

- **비전**: "대화로 완성하는, 나를 기억하는 멀티포맷 일기."
- **1순위 (북극성) — 일기 품질·다양성**: 결과물(일기) 자체가 *"돈 내고 또 보고 싶은"* 수준. 포맷·문체·길이·이미지·구조까지. → 상세 §8.
- **2순위 — 기억 기반 연속성**: 나를 기억하고 과거를 이어가는 동반자. → 상세 §4-A.
- **플랫폼**: 모바일(앱스토어) 우선. **방식**: 그린필드 풀 리라이트(스택 포함).

---

## 2. PoC에서 확정된 설계 원칙 (v2도 계승)

PoC 4라운드 테스트로 검증된, 흔들지 않을 원칙들:

1. **사실 날조 금지** — 유저가 말/확인한 것만. 감정·사건 순서·외부사실(날씨 등)을 지어내지 않는다. (단, 소설 포맷은 *표현*의 문학적 윤색만 허용, 사실관계는 동일 가드)
2. **감상은 추측-확인으로 끌어낸다** — "그때 ○○했겠다, 그치?"로 유저가 가볍게 긍정/수정 → 그것만 일기에 사용.
3. **시간·맥락 인지** — 현재 시각/하루 진행도 인지, 회고 강요 금지, "저녁에 다시" 제안.
4. **조급하게 끝내지 않기** — 모호하지만 의미 있는 실마리는 한 겹 더 묻되, 유저가 원치 않으면 넘어감.
5. **말투 일관(반말) + 리듬 다양** — 매 턴 질문 강제 금지, 진짜 반응.
6. **사진 = 작성칸에서 텍스트와 함께 전송 + AI가 실제로 봄(vision) + 일기 본문에 인라인 배치.**
7. **포맷별 톤** — 일반(담백)/신문(기사)/소설(윤색 허용).
8. **초안 → 수정 모드** — 생성 후 자유 지시로 다시 쓰기. 🔧 PoC 완료
9. **비용 직시** — Gemini 3 Flash가 품질·가격 압도(세션 ~₩30~70). LLM은 사업 비용의 푼돈, GPT/Claude는 가성비 탈락.

## 4. v2가 풀어야 할 핵심 미해결 문제

### A. 세션 간 기억 ★ (2순위 북극성)
- **문제**: 매 대화가 백지 → "아이 있어요?"를 또 물음. "어제 데모데이 어떻게 됐어?"를 못 함.
- **방식 확정**: 구조화 프로필 + 벡터검색 **혼합** (Postgres + pgvector).

**데이터 모델**
- `user_profile_fact` — 지속 사실. `{ userId, category(가족·직업·건강·관심사·진행중일 등), content, confidence, sourceMessageId, updatedAt }`. 수정/삭제 가능, 갱신 시 이력.
- `episodic_memory` — 날짜별 요약. `{ userId, date, summary, mood, embedding, conversationId }`.
- `diary` (+ `embedding`) — 과거 일기 보관 + 의미검색.
- 원본 대화/메시지 보존.

**쓰기(추출) 경로** — 일기 생성/대화 종료 시 **추출 단계**(LLM):
- (a) 새 지속 사실 → 프로필 추가/갱신 (b) 그날 에피소드 요약+무드 저장 + embedding.
- 원칙: **유저가 명시한 것만**(창작 금지 §2-1 동일). 모순 시 최신 우선 + 이력. confidence 부여.

**읽기 경로**
- **대화 시작**: 프로필 핵심 + 최근 N개 에피소드 요약을 system에 주입.
- **대화 중**: 관련 과거 기억을 **벡터검색**으로 회수(툴 `recallMemory`). 예: "지난번 그 프로젝트" → 관련 일기/에피소드.
- **일기 생성**: 연속성 표현("어제 시작한 …을 오늘 마무리")에 관련 기억 주입.

**프라이버시·UX**: 계정 전체 삭제는 출시에 포함(스토어 필수). **기억 개별 열람·수정·삭제·"기억하지 마" 토글은 후순위**(출시 후 별도). 그때까진 추출을 보수적으로(명시 사실만).
**리스크**: 잘못된 사실 고착(프로필 환각) → confidence·출처표시로 완화, 개별 수정 UI 전까진 추출 기준을 엄격히.

### B. 대화 깊이 · 인터뷰 전략 (1순위 품질의 절반)
PoC는 "공감+질문"을 LLM 자율에 맡김 → 모델 의존·조기종료. v2는 **수집 상태를 명시적으로 관리**한다.

**명시적 phase 상태머신**: `greeting → gather → (photo) → enough? → offer`
- **greeting**: 메모리(프로필·최근 에피소드)·시각·날씨 주입한 인사.
- **gather**: 포맷별 `수집 목표 체크리스트`를 상태로 들고, **매 턴 "무엇이 채워졌나"를 평가** → 빈 항목 우선 질문. (LLM 자율 질문이 아니라 상태 기반)
  - 예(일반): 주요 사건 / 시간 흐름 / 감정·생각 / 기억에 남는 순간.
  - **깊이 vs 넘어가기**: 모호하지만 의미 있는 실마리는 한 겹 더(§2-4), 꺼리면 넘어감.
  - **감상은 추측-확인**으로 채움(§2-2). "감정·생각" 항목은 *유저 확인*이 있어야 충족 처리.
- **enough?**: 체크리스트 충분 + 감상 확보 시 `offerDiary` 신호(조기종료 방지).
- 구현: 턴마다(또는 주기) 구조화 출력으로 수집 상태 갱신 → 다음 질문 선택. 메모리 회수는 `recallMemory` 툴.

### C. 포맷별 페르소나 심화
- 일반/신문/소설의 인터뷰 방식·질문 종류·일기 산출이 더 또렷하게 달라지도록. (기자는 육하원칙, 소설가는 감각/내면 등)

### D. 멀티 사진 / 순서
- 한 메시지에 여러 장, 일기 내 다중 배치, 순서/페어링.

### E. 비용 · 지연 · 신뢰성
- 프롬프트 캐싱, 캡션 생성 지연/통합으로 요청 절감, **무료 쿼터 한도/폴백 모델**, 재시도·사용자 안내.

### F. 평가 루프 (품질을 "측정 가능"하게)
- PoC의 **피드백 기능**을 자산화. 품질은 측정 없으면 못 올린다.
- **골든셋**: 대표 대화 시나리오 N개(짧은/긴/사진많은/감정적은 등) × 3포맷. 각 일기에 **체크리스트**:
  - 창작 없음(유저 말한 것만) / 시간순서 정확 / 포맷다움 / 감정 진정성 / 길이 적정 / 이미지 배치.
- **회귀**: 프롬프트·모델 변경 시 골든셋 재생성 → **LLM-as-judge + 체크리스트**로 before/after 비교.
- **운영 지표**: 창작율, 순서오류율, **유저 수정 횟수(낮을수록↑)**, 별점/유지율.

## 5. 아키텍처 스케치 (초안 — 논의용, 확정 아님)

```
[Client]  대화 UI · 일기 뷰 · 수정 · 피드백
   │
[Conversation Agent]  ── phase: greet → gather → (photo) → enough? → draft → revise
   │   ├─ reads:  UserProfile + 관련 과거 일기/에피소드 (시작 시 주입)
   │   ├─ tools:  requestPhoto, offerDiary, (new) recallMemory(검색), saveFact
   │   └─ vision: 첨부 이미지 인식
   │
[Memory Layer]
   ├─ UserProfile (durable facts, 수정/삭제 가능)
   ├─ Episodic memory (날짜별 사건·감정 요약)
   └─ Diary archive (+ 검색: 키워드/임베딩)
   │
[Diary Generator]  포맷별 프롬프트 · 시간순서 가드 · 이미지 인라인 · revise
   │
[Eval/Feedback]  피드백 수집 → 골든셋 → 회귀 비교
```

**모델 전략**: 기본 Gemini 3 Flash(품질·가격), vision 동일 모델, 폴백/쿼터 대응 ❓.

## 6. 결정 사항 (Decisions)

**확정됨 ✅**
1. **범위**: 그린필드 풀 리라이트(스택 포함).
2. **북극성**: ① 일기 품질·다양성 ② 기억 기반 연속성.
3. **플랫폼**: 모바일(앱스토어) 우선.
4. **모바일**: React Native CLI + TypeScript (Expo 제외).
5. **다양성**: 포맷 확장 중심(조절 노브 후순위).
6. **생성 방식**: 대화 → 단일 생성 → 수정(횟수 제한).

**확정됨 ✅ (이어서)**
7. **백엔드**: 직접 Node/Nest 서버. **DB**: Postgres + pgvector.
8. **메모리**: 구조화 프로필 + 벡터검색 혼합. → §4-A 상세
9. **1차 포맷**: 기존 3개(일반/신문/소설)로 시작, 시스템은 확장 가능하게. (포맷 추가는 출시 후)

**확정됨 ✅ (이어서)**
10. **수정 정책**: per-diary 무료 3회 / 구독 10회. 비용보다 기대치 관리용 캡.
11. **agent 구현**: 직접 오케스트레이션(명시적 phase 상태 + 툴) + 일기 생성은 AI SDK 단일 호출. 무거운 프레임워크 미사용.
12. **계정**: 소셜 로그인 = **Apple(iOS 필수) + Google + Kakao(한국)** + 서버 저장. 계정 삭제는 제공(스토어 필수). **기억 개별 열람·수정·삭제 UI는 후순위(출시 후 별도).**

## 8. 북극성 상세 — 일기 품질·다양성 (1순위)

두 축으로 나눠 설계한다.

### 8.1 품질 (Quality) — 한 편의 일기가 얼마나 좋은가
- **사실 충실 + 감정 진정성**: 창작 금지(§2-1), 감상은 확인된 것만(§2-2).
- **문체·리듬**: 보고서체 탈피, 그 순간의 장면과 유저 어휘를 살림. 포맷별 톤.
- **구조·길이**: 도입–전개–여운, 하루 분량에 맞는 적정 길이(너무 짧음/나열 금지).
- **이미지·시간순서**: 인라인 배치(§2-6), 순서 단정 금지(§2-1).

**품질 생성 방식 ✅ 확정**: **대화 → 일기 생성(단일 패스) → 유저 수정(횟수 제한)**.
- 품질은 세 곳에서 확보: ① **대화 인터뷰 품질**(필요 정보·감상 충분히 수집) ② **포맷별 생성 프롬프트 + 가드**(창작 금지·시간순서·문체) ③ **제한된 수정 루프**(유저가 직접 고침).
- 내부 다단계(plan→self-critique→다듬기)는 **선택적 후순위** — 품질이 부족하면 그때 도입.
- **❓ 수정 정책**: per-diary 수정 횟수 상한(예: 무료 N회 / 유료 더). 1회 ≈ ₩수십(Gemini)이라 비용보단 *남용·기대치 관리*용 캡.

### 8.2 다양성 (Diversity) ✅ — **포맷 확장 중심**
- 방향 확정: **유저 조절 노브보다 "포맷 개수"로 다양성**을 준다. (노브는 후순위)
- **포맷 시스템을 확장 가능하게 설계**: 한 포맷 = `{ 페르소나, 인터뷰 수집목표, enough 기준, 생성 프롬프트, 예시 }`. **신규 포맷 추가가 데이터/프롬프트만으로** 되도록(코드 변경 최소).
- **후보 로스터** (1차 출시 세트 ❓):
  - 계승: 일반 / 신문 / 소설

## 10. 대화→일기 품질 파이프라인 (1순위 통합 뷰)

```
대화 시작 ── 메모리 주입(프로필+최근 에피소드) + 시각/날씨
   │
gather ── 수집 체크리스트 상태머신 (빈 항목 우선 질문, 감상=추측확인, recallMemory)
   │        └ 사진: 작성칸 동반 전송 + AI 비전
   │
enough? ── 체크리스트+감상 충분 → offerDiary
   │
일기 생성(단일) ── 입력: transcript + 사진캡션 + 관련 기억(연속성) + 포맷정의
   │               가드: 창작금지·시간순서·문체·이미지 인라인
   │
후처리 ── ① 메모리 추출(프로필 갱신 + 에피소드 요약, 명시 사실만)
   │       ② 일기 embedding 저장
   │
수정 루프 ── 유저 지시로 재작성 (무료 3/구독 10회)
   │
평가 ── 골든셋 회귀 + 운영지표(창작율·순서오류·수정횟수·별점)
```

**품질의 3대 레버**: ① gather 인터뷰(상태머신) ② 생성 프롬프트+가드 ③ 제한된 수정.
**연속성**: 생성/대화에 관련 기억 주입 → "어제 시작한 …을 오늘 마무리".

## 11. 빌드 로드맵 (그린필드·솔로)

> 그린필드 = 새 레포에서 처음부터. PoC의 설계·프롬프트는 자산으로 이식하되 코드는 새로.
> 원칙: **북극성 #1(품질)을 단일 유저로 먼저 완성 → 계정 → #2 기억 → 수익화 → 출시.** 평가는 M1부터 병행.

| 단계 | 내용 | "끝" 기준 |
|---|---|---|
| **M0 기반** | RN CLI+TS 앱 · Node/Nest API · Postgres+pgvector · 공유 타입 · 환경/시크릿 · 최소 CI | 빈 앱이 API와 핸드셰이크 |
| **M1 핵심 대화→일기 ★** | phase 상태머신·수집 체크리스트·감상확인 / 3포맷 생성+가드 / 사진 동반전송+비전+인라인 / 수정(제한) / 시각·날씨. **메모리·계정 없이** | 한 유저가 대화→3포맷 일기→수정까지 (PoC의 "제대로 된" 재구현) |
| **M-Eval (병행)** | 골든셋 + 회귀 + 운영지표 (M1부터 같이) | 프롬프트 바꾸면 품질 비교가 됨 |
| **M2 계정·영속** | Apple/Google/Kakao 로그인 · 유저별 대화·일기 저장 · 계정 삭제 | 로그인해서 내 일기가 쌓임 |
| **M3 기억 (#2)** ✅구현 | 프로필+에피소드+벡터 · 추출(쓰기)·주입/회수(읽기) · 연속성 | "어제 …어떻게 됐어?"가 됨 |
| **M4 구독·수익화** | 앱스토어 IAP · 무료/유료 게이팅(수정 횟수 등) · 무료체험 | 결제하고 유료 기능 열림 |
| **M5 출시 준비** | 온보딩 · 지역가격 · 신뢰성(쿼터·폴백·재시도·에러 UX) · 개인정보·스토어 심사 · 모니터링 | 스토어 제출 |

핵심: **M0~M1로 "품질 코어"를 빨리 세우는 것.** 거기서 다시 본인이 써보며 PoC처럼 피드백 루프를 돌리면 됨. (시간 추정은 생략 — 범위가 더 중요)

### M3 구현 메모 (2026-06-09, S4 전 선제 구현)
풀 스택으로 구현. 임베딩 = OpenAI `text-embedding-3-small`(1536). pgvector는 TypeORM `synchronize`가 모르는 타입이라 **별도 raw 테이블 `memory_embedding(vector(1536))`** 로 격리(부트스트랩에서 멱등 생성, 일기 재생성 대비 `(ownerType,ownerId)` upsert).
- **엔티티**: `user_profile_fact`(category·content·confidence·supersededAt 이력) · `episodic_memory`(date·summary·mood, 대화당 1건 upsert).
- **쓰기**(`MemoryService.onDiaryComplete`): 일기 생성 후처리 — LLM 구조화 추출(`generateObject`, 명시 사실만·보수적) → 프로필 dedupe-insert + 에피소드 + 에피소드/일기 임베딩. best-effort(실패 비차단). `memory_extraction` step으로 비용 트레이싱.
- **읽기**: greeting·일기 생성에 `buildContext`(프로필+최근 에피소드) 주입 / 대화 중 `recallMemory` 툴이 pgvector `<=>` 의미검색.
- **검증**: api 테스트 128개·커버리지 100% 유지. 라이브 — 일기 재생성→프로필 2건·에피소드·임베딩 저장 / 새 대화 greeting 프롬프트에 기억 주입 확인.
- **남은 것(후순위)**: 기억 개별 열람·수정·삭제 UI(출시 후) · "기억하지 마" 토글 · 모순 자동 supersede(현재는 exact-dedupe만).

### 11.1 M0 — 기반 스캐폴딩 (착수 단위)
